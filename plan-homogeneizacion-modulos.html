1. Contexto y Objetivos

Por que se necesita este plan y que se espera conseguir.

Situacion actual

El ecosistema de proyectos de CODANOR en ~/Proyectos/ contiene 17 modulos activos (mas PLATAFORMA_OpenCode como base, modulo 0), organizados en 3 capas segun el diagrama Estructura_PROYECTOS.pdf: Introduccion, Elaboracion y Apoyo.

Cada modulo deberia tener 4 artefactos obligatorios: AGENTS.md, Memory.md, una carpeta Prompt/ y una carpeta Manuales/. Tras una auditoria exhaustiva, se ha detectado que solo 7 de los 18 modulos cumplen completamente este estandar.

Objetivo del plan

Homogeneizar

Que los 17 modulos tengan la misma estructura base: AGENTS.md, Memory.md, Prompt/, Manuales/, opencode.json, .gitignore.

Integrar clientes

Establecer un patron estandar para que cada modulo pueda consumir clientes_codanor.json de forma autonoma y opcional.

Garantizar autonomia

Cada modulo debe funcionar de forma independiente, sin depender de la existencia de otros modulos ni de rutas fijas.

Que NO incluye este plan

  • Crear el modulo GeNIS2 (modulo 10) — se deja para una fase posterior.
  • Inicializar .git y subir a GitHub los proyectos sin repositorio.
  • Corregir bugs de SonarCloud (innerHTML, CORS, etc.).
  • Instalar Node.js, Homebrew o gh en el equipo.
  • Automatizar backups con cron.

4. Principios de Autonomia

Reglas inviolables que garantizan que cada modulo puede funcionar, desarrollarse y mantenerse de forma completamente independiente.

Principio 1: Standalone por defecto

Cada modulo arranca, se ejecuta y es util sin que exista clientes_codanor.json ni ningun otro fichero de otro proyecto. Los datos compartidos son un enriquecimiento opcional, nunca un requisito.

Principio 2: Cero imports cruzados

Ningun modulo referencia directamente la carpeta de otro modulo. El unico punto de intercambio es ~/Proyectos/clientes_codanor.json (raiz compartida), y se accede siempre con degradacion elegante (si falta, warning — no error).

Principio 3: Repositorio independiente

Cada carpeta es (o sera) un repositorio Git independiente. Puede clonarse en cualquier maquina, en cualquier ruta, y funcionar. El codigo nunca asume una ruta absoluta — las rutas externas se configuran via variables de entorno con fallback.

Principio 4: Datos locales, nunca remotos

El codigo de la aplicacion lee siempre una copia local del fichero de clientes (e.g. ./Datos/clientes/clientes_codanor.json), nunca directamente de ~/Proyectos/. Solo el script iniciar.sh conoce la ruta de origen.

Principio 5: Proteccion RGPD integrada

Todo fichero con datos de clientes reales esta excluido de Git via .gitignore. Nunca se commitea. Nunca se sube a GitHub ni a ninguna nube publica.

Test de autonomia

Cada modulo debe pasar estas 3 pruebas para considerarse autonomo:

PruebaVerificacion
Standalone git clone <repo> /tmp/test && cd /tmp/test && ./iniciar.sh funciona (con warning de clientes, pero arranca)
Sin raiz El modulo funciona en cualquier ruta, no solo en ~/Proyectos/
Sin hermanos El modulo NO referencia ningun otro modulo por nombre ni ruta

Patron estandar de consumo

Todos los modulos que consuman el fichero de clientes deben seguir este patron de 4 capas de proteccion:

  1. Capa 1: iniciar.sh (script de arranque, opcional)
    Intenta copiar el fichero desde la raiz. La ruta es configurable via variable de entorno CLIENTES_JSON. Si falla, imprime warning pero NO aborta. 
    CLIENTES_JSON="${CLIENTES_JSON:-$HOME/Proyectos/clientes_codanor.json}"
if [ -f "$CLIENTES_JSON" ]; then
    cp "$CLIENTES_JSON" "$DIR/Datos/clientes/clientes_codanor.json"
    echo "[OK] Clientes actualizados"
else
    echo "[WARN] Sin datos de clientes. Funcionalidad limitada."
fi
  2. Capa 2: Codigo de la aplicacion
    Lee SIEMPRE la copia local: ./Datos/clientes/clientes_codanor.json. NUNCA referencia ~/Proyectos/ directamente desde el codigo fuente. Si no encuentra el fichero, muestra un boton "Importar clientes" o simplemente oculta el selector de cliente.
  3. Capa 3: .gitignore
    Excluye siempre: Datos/clientes/clientes_codanor.json, clientes_codanor.json, *.sqlite, .env. El fichero de clientes nunca se commitea.
  4. Capa 4: AGENTS.md
    Documenta la dependencia opcional en una seccion "Datos compartidos": que es, de donde viene, que pasa si falta.

Caso especial: Plantillas_ENS_ISO

clientes_codanor.json != clientes_guardados.json

Plantillas_ENS_ISO tiene su propio fichero clientes_guardados.json con un esquema completamente distinto (campos como cl-email-dpo, cl-admin-user, cl-sharepoint-url) orientado a rellenar placeholders en plantillas documentales .docx.

NO se fusionan. Son complementarios: clientes_codanor.json es el directorio de contacto (quien es el cliente), clientes_guardados.json son datos de formulario para generar documentos. Se documenta en AGENTS.md y opcionalmente se permite pre-rellenar campos comunes (nombre, CIF) desde clientes_codanor.json como import opcional.

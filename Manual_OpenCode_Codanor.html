1. Introduccion a OpenCode Actualizado

Fundamentos de OpenCode, la herramienta de codificacion con IA de codigo abierto que utilizamos en CODANOR.

Que es OpenCode

OpenCode es un agente de codificacion con inteligencia artificial de codigo abierto desarrollado por Anomaly. Permite interactuar con modelos de lenguaje (LLM) para explorar, modificar y gestionar proyectos de software. Esta disponible en multiples modalidades: terminal (TUI), CLI, aplicacion de escritorio, interfaz web y extension de IDE.

Version actual
OpenCode se encuentra en la version 1.3.x (Marzo-Abril 2026). Soporta multiples proveedores de IA incluyendo Anthropic (Claude), OpenAI (GPT), Google (Gemini), y muchos mas a traves de OpenCode Zen.

Modalidades de uso

TUI (Terminal)

Interfaz interactiva en la terminal. Es el modo principal y el mas completo. Se lanza con opencode.

CLI (Linea de comandos)

Ejecucion directa sin interfaz interactiva. Ideal para scripts y CI/CD: opencode run "mensaje".

Desktop App

Aplicacion de escritorio nativa con interfaz grafica. Ejecuta un servidor OpenCode local en segundo plano.

Web / IDE

Interfaz web accesible desde el navegador (opencode web) o extension integrada en tu editor favorito.

2. Agentes y Modos de Operacion Nuevo

OpenCode utiliza un sistema de agentes especializados. Comprender como funcionan es clave para usar la herramienta de forma eficiente.

Agentes primarios

Los agentes primarios son los asistentes principales con los que interactuas directamente. Se alternan con la tecla Tab.

Build (por defecto)

Todas las herramientas habilitadas. Es el agente de trabajo habitual para desarrollo: lee, escribe, ejecuta comandos y modifica archivos.

Plan

Agente restringido para planificacion y analisis. Las operaciones de edicion y bash requieren aprobacion (ask). Ideal para revisar antes de implementar.

Plan Mode no es 100% de solo lectura
A diferencia de versiones anteriores, Plan Mode ahora usa un sistema de permisos. Las herramientas de edicion y bash estan configuradas como "ask", lo que significa que te pedira aprobacion antes de ejecutarlas. No las bloquea completamente.

Subagentes

Los subagentes son asistentes especializados que los agentes primarios pueden invocar automaticamente, o que puedes llamar manualmente con @nombre: 

@general ayudame a buscar esta funcion
@explore encuentra todos los archivos de test

General

Agente de proposito general para tareas complejas y multi-paso. Tiene acceso completo a herramientas, puede modificar archivos. Ideal para ejecutar trabajo en paralelo.

Explore

Agente rapido de solo lectura para explorar codigo. No puede modificar archivos. Ideal para buscar patrones, encontrar definiciones o responder preguntas sobre el codigo.

Agentes ocultos (sistema)

OpenCode tambien tiene agentes internos que se ejecutan automaticamente:

  • Compaction — Comprime el contexto largo en un resumen mas pequeno
  • Title — Genera titulos cortos para las sesiones
  • Summary — Crea resumenes de sesion

Navegar entre sesiones de subagentes

AccionKeybind por defecto
Entrar en sesion hijoLeader+↓
Siguiente sesion hijo→
Anterior sesion hijo←
Volver al padre↑

Crear agentes personalizados

Puedes crear agentes propios de dos formas:

1. Mediante archivos Markdown

Crea un archivo .md en .opencode/agents/ (proyecto) o ~/.config/opencode/agents/ (global):

# .opencode/agents/review.md
---
description: Revisa codigo buscando problemas
mode: subagent
model: anthropic/claude-sonnet-4-5
temperature: 0.1
permission:
  edit: deny
  bash:
    "*": ask
    "git diff": allow
    "grep *": allow
---
Eres un revisor de codigo. Enfocate en:
- Calidad y buenas practicas
- Bugs potenciales
- Rendimiento
- Seguridad

2. Mediante opencode.json

{
  "agent": {
    "code-reviewer": {
      "description": "Revisa codigo en busca de problemas",
      "mode": "subagent",
      "model": "anthropic/claude-sonnet-4-5",
      "permission": {
        "edit": "deny",
        "bash": "ask"
      }
    }
  }
}

3. Comando interactivo

opencode agent create

Este comando guiado te preguntara donde guardar el agente, su descripcion, herramientas y generara el archivo automaticamente.

Consejo
Usa default_agent en opencode.json para definir que agente se usa por defecto al iniciar: 
{ "default_agent": "build" }

3. Permisos y Seguridad Nuevo

Sistema granular de permisos para controlar que acciones puede realizar el agente sin tu aprobacion.

Acciones disponibles

AccionDescripcion
"allow"Ejecutar sin pedir aprobacion
"ask"Pedir aprobacion antes de ejecutar
"deny"Bloquear la accion completamente

Configuracion basica

{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "*": "ask",
    "bash": "allow",
    "edit": "deny",
    "read": "allow"
  }
}

Permisos granulares con wildcards

Para la herramienta bash, puedes definir reglas por comando. La ultima regla que coincida gana:

{
  "permission": {
    "bash": {
      "*": "ask",
      "git *": "allow",
      "git push *": "deny",
      "npm *": "allow",
      "rm *": "deny",
      "grep *": "allow"
    },
    "edit": {
      "*": "allow",
      "*.env": "deny",
      "*.env.*": "deny"
    }
  }
}

Permisos disponibles

PermisoControla
readLectura de archivos
editTodas las modificaciones (edit, write, patch, multiedit)
bashComandos de terminal
glob / grep / listBusqueda de archivos y contenido
taskInvocacion de subagentes
skillCarga de skills
webfetch / websearchAcceso a internet
external_directoryAcceso a rutas fuera del proyecto (default: ask)
doom_loopDetecta cuando una herramienta se repite 3 veces con misma entrada (default: ask)

Permisos por agente

Cada agente puede tener sus propios permisos que sobreescriben los globales:

{
  "permission": { "bash": "ask" },
  "agent": {
    "build": {
      "permission": {
        "bash": {
          "*": "ask",
          "git status *": "allow"
        }
      }
    }
  }
}

Que hace "Ask"

Cuando OpenCode pide aprobacion, tienes tres opciones:

  • once — Aprobar solo esta vez
  • always — Aprobar futuros patrones similares (durante la sesion)
  • reject — Rechazar la solicitud
Valores por defecto
Por defecto, la mayoria de permisos estan en "allow". Los archivos .env tienen lectura denegada por defecto. doom_loop y external_directory estan en "ask".

4. Configuracion Avanzada Nuevo

Todo lo que necesitas saber sobre los archivos de configuracion de OpenCode, su estructura y opciones.

Archivos de configuracion

ArchivoPropositoUbicacion
opencode.jsonConfig del servidor/runtimeProyecto o ~/.config/opencode/
tui.jsonConfig de la interfaz TUIProyecto o ~/.config/opencode/
AGENTS.mdReglas e instrucciones del proyectoRaiz del proyecto
Formatos soportados
OpenCode soporta JSON y JSONC (JSON con comentarios). Usa la extension .jsonc para poder incluir comentarios.

Orden de precedencia

Las configuraciones se fusionan (no se reemplazan). El orden de prioridad (el ultimo gana):

  1. Config remota — Desde .well-known/opencode (defaults organizacionales)
  2. Config global — ~/.config/opencode/opencode.json
  3. Config custom — Variable OPENCODE_CONFIG
  4. Config de proyecto — opencode.json en la raiz del proyecto
  5. Directorios .opencode — Agentes, comandos, plugins
  6. Config inline — Variable OPENCODE_CONFIG_CONTENT

Opciones principales

Modelo y proveedor

{
  "$schema": "https://opencode.ai/config.json",
  "model": "anthropic/claude-sonnet-4-5",
  "small_model": "anthropic/claude-haiku-4-5",
  "provider": {
    "anthropic": {
      "options": {
        "timeout": 600000,
        "chunkTimeout": 30000
      }
    }
  }
}

small_model se usa para tareas ligeras como generar titulos de sesion.

Compaction automatica

{
  "compaction": {
    "auto": true,
    "prune": true,
    "reserved": 10000
  }
}
  • auto — Compactar automaticamente cuando el contexto se llena (default: true)
  • prune — Eliminar outputs antiguos de herramientas para ahorrar tokens (default: true)
  • reserved — Buffer de tokens reservado para la compactacion

Snapshots (Undo/Redo)

{
  "snapshot": false
}

Desactiva snapshots si el proyecto es muy grande y el indexado es lento. Sin snapshots no hay undo/redo.

Autoupdate

{
  "autoupdate": true
}

Usa "notify" para solo recibir aviso de nuevas versiones sin instalar automaticamente.

Variables en la configuracion

Puedes usar variables de entorno y contenido de archivos:

{
  "model": "{env:OPENCODE_MODEL}",
  "provider": {
    "openai": {
      "options": {
        "apiKey": "{file:~/.secrets/openai-key}"
      }
    }
  }
}

Instrucciones adicionales

{
  "instructions": [
    "CONTRIBUTING.md",
    "docs/guidelines.md",
    ".cursor/rules/*.md",
    "https://example.com/shared-rules.md"
  ]
}

Acepta rutas locales, globs y URLs remotas. Se combinan con tu AGENTS.md.

TUI (tui.json)

{
  "$schema": "https://opencode.ai/tui.json",
  "theme": "opencode",
  "keybinds": { "leader": "ctrl+x" },
  "scroll_speed": 3,
  "scroll_acceleration": { "enabled": true },
  "diff_style": "auto"
}

5. Servidores MCP Nuevo

Integra herramientas externas mediante el Model Context Protocol (MCP) para ampliar las capacidades de OpenCode.

Cuidado con el contexto
Cada servidor MCP anade herramientas al contexto del modelo. Demasiados servidores pueden exceder el limite de contexto rapidamente. Se selectivo con los que activas.

Servidor local

{
  "mcp": {
    "mi-servidor": {
      "type": "local",
      "command": ["npx", "-y", "mi-mcp-server"],
      "enabled": true,
      "environment": {
        "API_KEY": "{env:MI_API_KEY}"
      }
    }
  }
}

Servidor remoto

{
  "mcp": {
    "mi-servidor-remoto": {
      "type": "remote",
      "url": "https://mcp.ejemplo.com/mcp",
      "enabled": true,
      "headers": {
        "Authorization": "Bearer {env:MI_API_KEY}"
      }
    }
  }
}

OAuth automatico

OpenCode maneja OAuth automaticamente. Para servidores que lo requieren:

# Autenticar manualmente
opencode mcp auth nombre-servidor

# Listar servidores y estado de auth
opencode mcp list

# Eliminar credenciales almacenadas
opencode mcp logout nombre-servidor

Ejemplos populares

Sentry

Monitoreo de errores. "type": "remote", "url": "https://mcp.sentry.dev/mcp" con OAuth.

Context7

Busqueda en documentacion. "url": "https://mcp.context7.com/mcp". Gratis con rate limits.

Grep (Vercel)

Buscar snippets en GitHub. "url": "https://mcp.grep.app". Sin auth requerida.

Tu propio MCP

Crea MCPs locales con cualquier lenguaje. Usa "type": "local" con el comando de inicio.

Gestion por agente

Desactiva un MCP globalmente y activalo solo para un agente especifico:

{
  "mcp": {
    "sentry": { "type": "remote", "url": "...", "enabled": true }
  },
  "tools": { "sentry_*": false },
  "agent": {
    "debug": { "tools": { "sentry_*": true } }
  }
}

6. Comandos Personalizados y Skills Nuevo

Automatiza tareas repetitivas con comandos personalizados y proporciona instrucciones reutilizables mediante Agent Skills.

Comandos personalizados

Crea archivos Markdown en .opencode/commands/ (proyecto) o ~/.config/opencode/commands/ (global):

# .opencode/commands/test.md
---
description: Ejecutar tests con cobertura
agent: build
model: anthropic/claude-haiku-4-5
---
Ejecuta la suite de tests completa con reporte de cobertura.
Muestra los tests fallidos y sugiere correcciones.

Usa el comando escribiendo /test en la TUI.

Argumentos

# .opencode/commands/component.md
---
description: Crear un componente React
---
Crea un componente React llamado $ARGUMENTS con TypeScript.
Incluye tipado correcto y estructura basica.

# Uso: /component Button

Tambien puedes usar $1, $2, $3 para argumentos posicionales.

Output de comandos shell

# .opencode/commands/review-changes.md
---
description: Revisar cambios recientes
---
Commits recientes:
!`git log --oneline -10`

Revisa estos cambios y sugiere mejoras.

Referencias a archivos

Revisa el componente en @src/components/Button.tsx.
Busca problemas de rendimiento.

Agent Skills

Skills son instrucciones reutilizables que los agentes cargan bajo demanda. Se definen como archivos SKILL.md:

# .opencode/skills/git-release/SKILL.md
---
name: git-release
description: Crear releases consistentes y changelogs
---
## Que hago
- Redacto notas de release desde PRs fusionados
- Propongo un version bump
- Proporciono el comando `gh release create`

## Cuando usarme
Usa este skill al preparar un release.

Ubicaciones de skills

  • .opencode/skills/<nombre>/SKILL.md (proyecto)
  • ~/.config/opencode/skills/<nombre>/SKILL.md (global)
  • .claude/skills/<nombre>/SKILL.md (compatibilidad Claude Code)
Requisitos del nombre
El nombre del skill debe ser 1-64 caracteres, alfanumerico en minusculas con guiones simples como separador. Debe coincidir con el nombre del directorio.

7. OpenCode Zen Nuevo

Proveedor curado de modelos verificados por el equipo de OpenCode. Opcional pero recomendado para garantizar la mejor calidad.

Que es Zen

OpenCode Zen es un gateway de IA que da acceso a una seleccion de modelos que han sido probados y verificados para funcionar bien como agentes de codigo. El equipo de OpenCode trabajo con los proveedores para asegurar la mejor configuracion.

Como activarlo

  1. Ve a opencode.ai/auth y crea una cuenta
  2. Anade datos de facturacion y copia tu API key
  3. En la TUI ejecuta /connect, selecciona OpenCode y pega la key
  4. Usa /models para ver los modelos disponibles

Modelos destacados

ModeloIDUso recomendado
Claude Opus 4.6opencode/claude-opus-4-6Tareas complejas, maximo rendimiento
Claude Sonnet 4.5opencode/claude-sonnet-4-5Desarrollo general, equilibrio calidad/velocidad
Claude Haiku 4.5opencode/claude-haiku-4-5Tareas rapidas, bajo coste
GPT 5.4opencode/gpt-5.4Desarrollo general con OpenAI
Gemini 3.1 Proopencode/gemini-3.1-proContexto largo, analisis
Modelos gratuitosVariosMiniMax, MiMo, Qwen, Big Pickle (temporales)

Equipos

Zen funciona para equipos: invita miembros, asigna roles (Admin/Member), controla que modelos estan disponibles y establece limites de gasto mensuales por miembro.

Bring Your Own Key (BYOK)

Si ya tienes keys de OpenAI o Anthropic, puedes usarlas en Zen. Los tokens se facturan directamente por el proveedor, no por Zen.

Sin lock-in
Zen es completamente opcional. Puedes usar cualquier proveedor directamente con OpenCode. Y puedes usar Zen con cualquier otro agente de coding.

8. Error de Dimensiones de Imagen

Explicacion y resolucion del error relacionado con imagenes demasiado grandes en la API del modelo.

El mensaje de error

messages.104.content.1.image.source.base64.data:
At least one of the image dimensions exceed max allowed
size for many-image requests: 2000 pixels

Por que ocurre

EscenarioLimite por dimension
Una sola imagenHasta ~8000 px
Multiples imagenesMaximo 2000 px

Cuando la conversacion acumula varias imagenes, se activa el limite mas estricto.

Soluciones

  1. Redimensionar antes de enviar: ni ancho ni alto deben superar 2000 px
  2. /compact para eliminar imagenes del historial
  3. Compaction automatica: activa "compaction": {"auto": true} en config para prevenir acumulacion
  4. Nueva sesion (/new) para empezar limpio

Redimensionar en macOS

# Usando sips (nativo)
sips --resampleWidth 1800 imagen.png

# Usando ImageMagick
convert imagen.png -resize 1800x imagen_small.png

9. Preservar el Historico del Proyecto Actualizado

Como mantener el contexto del proyecto entre sesiones usando mecanismos de persistencia.

AGENTS.md

El archivo AGENTS.md es la memoria permanente del proyecto. OpenCode lo lee al iniciar cada sesion.

/init
  • /init analiza tu repositorio y crea o actualiza AGENTS.md
  • Incluye: comandos de build/test, estructura, convenciones, quirks del proyecto
  • Siempre committealo a Git

Compatibilidad con Claude Code

OpenCode lee CLAUDE.md como fallback si no existe AGENTS.md. Tambien reconoce ~/.claude/CLAUDE.md para reglas globales.

# Desactivar compatibilidad Claude Code
export OPENCODE_DISABLE_CLAUDE_CODE=1

AGENTS.md global

Crea ~/.config/opencode/AGENTS.md para reglas personales que apliquen a todas las sesiones. No se commitea a Git.

Instrucciones externas

En opencode.json puedes referenciar archivos adicionales de instrucciones:

{
  "instructions": [
    "CONTRIBUTING.md",
    "docs/guidelines.md",
    ".cursor/rules/*.md",
    "packages/*/AGENTS.md"
  ]
}

/compact

Comprime la conversacion eliminando imagenes y detalles de herramientas, conservando el contexto esencial en texto.

Compaction automatica
Con "compaction": {"auto": true} en la config, OpenCode compacta automaticamente cuando el contexto se llena. Es la opcion recomendada.

Flujo recomendado

  1. /init — Crea AGENTS.md (memoria del proyecto)
  2. Configura compaction auto — En opencode.json
  3. Trabaja normalmente — Las imagenes y contexto se gestionan automaticamente
  4. /compact manual — Cuando notes lentitud o antes de tareas grandes
  5. Nueva sesion si es necesario — OpenCode lee AGENTS.md al iniciar

10. Procesos Bloqueados ("Pensando...")

Que hacer cuando OpenCode se queda en estado "Pensando" y no avanza.

Por que se bloquea

Cuando OpenCode ejecuta un comando como npm start o node server.js, ese proceso nunca termina porque queda escuchando conexiones. OpenCode espera a que finalice, creando un bloqueo infinito.

Solucion inmediata

AccionTecla
CancelarEsc
Interrumpir (si Esc no funciona)ctrl+c

Prevencion

# MAL:
"Levanta el servidor para probar"

# BIEN:
"Verifica que el codigo compila con: npm run build"

# O SI NECESITAS el servidor:
"Lanza el servidor en segundo plano con & y verifica
que arranco. No esperes a que termine."
Regla general
Nunca pidas a OpenCode que ejecute procesos que no terminan sin indicarle que los lance en segundo plano o que use un timeout.

11. Rendimiento y Lentitud Actualizado

Causas y soluciones cuando OpenCode responde lentamente.

Causas principales

Conversacion larga

Cada mensaje envia todo el historial al modelo. Solucion: /compact o activar compaction automatica.

Modelo pesado

Claude Opus es el mas lento. Usa /models para cambiar a Sonnet o Haiku para tareas simples.

Proyecto grande

Muchos archivos = busquedas lentas. Se especifico en las instrucciones.

Snapshots en repos grandes

Si el indexado de snapshots es lento, desactivalos: "snapshot": false en la config.

Soluciones

CausaSolucion
Historial largo/compact + "compaction": {"auto": true, "prune": true}
Modelo lento/models para cambiar a Sonnet/Haiku. Usa small_model en config
Proyecto grandeSe especifico. Usa @archivo para dar contexto exacto
Snapshots lentos"snapshot": false (pierdes undo/redo)
MCP excesivosDesactiva MCPs que no uses: "tools": {"mcp_*": false}
Consejo
Cuanto mas preciso seas en tus instrucciones, menos tiene que buscar OpenCode y mas rapido responde. Usa @archivo.ts para dar contexto directo.

12. Error "Tool Execution Aborted" Nuevo

Diagnostico y solucion del error "Ejecucion de la herramienta abortada".

Que significa

Este error aparece cuando OpenCode intenta usar una herramienta pero la operacion es cancelada antes de completarse.

Causas principales

1. Permisos insuficientes (Plan Mode)

Si estas en el agente Plan y el modelo intenta escribir archivos, la operacion requiere aprobacion (ask). Si se rechaza o si el agente no espera la aprobacion, la herramienta se aborta.

Solucion
Cambia al agente Build con Tab antes de pedir que se creen o modifiquen archivos. Verifica el indicador en la esquina inferior derecha de la TUI.

2. Contenido demasiado grande

Cuando el archivo a escribir excede los limites del mensaje de la API.

3. Historial de sesion demasiado largo

Ejecuta /compact para comprimir el historial.

4. Timeout de herramienta

El comando tardo demasiado. Aumenta el timeout en config si es necesario.

Solucion paso a paso

  1. Verifica que estas en Build Mode (mira la esquina inferior derecha)
  2. Si el error persiste, ejecuta /compact
  3. Si sigue fallando, inicia nueva sesion con /new
  4. Para archivos muy grandes, divide la operacion en partes mas pequenas

13. Troubleshooting General NuevoPrioritario

Guia completa de resolucion de problemas comunes en OpenCode.

Logs

14. Buenas Practicas Actualizado

Recomendaciones para trabajar de forma eficiente y segura con OpenCode.

Se especifico

Indica archivos concretos con @, nombres exactos y alcance del cambio. Menos ambiguedad = mas velocidad.

Activa compaction auto

Mantiene AGENTS.md

Es la memoria persistente. Usa /init para crearlo y committealo a Git.

Usa Plan antes de Build

Cambia a Plan (Tab) para revisar la estrategia antes de implementar. Luego vuelve a Build.

Configura permisos

Bloquea operaciones peligrosas como rm * y git push. Usa "ask" para revisarlas.

Elige el modelo adecuado

Opus para lo complejo, Sonnet para desarrollo general, Haiku para tareas rapidas.

Crea comandos custom

Usa subagentes

Usa @explore para buscar en el codigo y @general para tareas multi-paso.

Redimensiona imagenes

Evita procesos sin fin

No pidas levantar servidores directamente. Usa build o check para verificar.

Usa /undo con confianza

Si algo no quedo bien, /undo revierte cambios (requiere que el proyecto sea un repo Git).

Se selectivo con MCPs

Cada MCP consume contexto. Solo activa los que realmente necesites para la tarea actual.

Exportar sesiones individuales

Desde la TUI

/export     # Exporta la sesion actual a Markdown (abre en tu editor)

Desde la CLI

# Listar todas las sesiones
opencode session list

# Exportar sesion especifica a JSON
opencode export <sessionID> > backup-sesion.json

# Exportar interactivamente (seleccionar de la lista)
opencode export

Importar una sesion

# Desde archivo JSON local
opencode import backup-sesion.json

# Desde una URL compartida
opencode import https://opncd.ai/s/abc123

Backup completo con script

Hemos creado el script backup-opencode.sh que automatiza todo el proceso. Incluido en este repositorio.

Uso del script

ComandoFuncion
./backup-opencode.shBackup completo (sesiones + config + cache)
./backup-opencode.sh --sessionsSolo sesiones y mensajes
./backup-opencode.sh --configSolo archivos de configuracion
./backup-opencode.sh --exportExportar cada sesion a JSON individual
./backup-opencode.sh --listListar backups existentes
./backup-opencode.sh --restore <archivo>Restaurar un backup
./backup-opencode.sh --cleanEliminar backups antiguos

Automatizar con cron

# Editar crontab
crontab -e

# Anadir esta linea (backup diario a las 2am)
0 2 * * * /ruta/a/backup-opencode.sh >> /ruta/a/backup.log 2>&1
Variable de entorno
Por defecto los backups se guardan en ~/Documents/opencode-backups/. Cambialo con: export OPENCODE_BACKUP_DIR="/otra/ruta"

Backup manual rapido (sin script)

# Backup completo con tar
tar -czf ~/opencode-backup-$(date +%Y%m%d).tar.gz \
  -C "$HOME" \
  .local/share/opencode/ \
  .config/opencode/

# Solo sesiones (mas ligero)
tar -czf ~/opencode-sessions-$(date +%Y%m%d).tar.gz \
  -C "$HOME" .local/share/opencode/storage/

No olvidar: archivos de proyecto en Git

Estos archivos son parte de tu proyecto y deberian estar versionados:

git add AGENTS.md opencode.json .opencode/
git commit -m "Add OpenCode project configuration"
No committear secretos
El archivo auth.json contiene API keys. Nunca lo subas a Git. Anadelo a .gitignore si lo copias al proyecto.

Restaurar un backup

  1. Con el script: ./backup-opencode.sh --restore backup.tar.gz (crea un backup de seguridad antes de sobreescribir)
  2. Manualmente: tar -xzf backup.tar.gz -C $HOME
  3. Importar sesion individual: opencode import sesion.json

17. Analisis de Codigo con SonarQube Actualizado

Guia completa y reproducible para configurar el analisis de calidad y seguridad del codigo con SonarQube Cloud, incluyendo la integracion con OpenCode y el panel de seguimiento local.

Que es SonarQube y por que usarlo

SonarQube Cloud (de SonarSource) es la herramienta lider en analisis estatico de codigo. Detecta automaticamente:

TipoQue detectaEjemplo
VulnerabilidadesProblemas de seguridad explotablesXSS, inyeccion SQL, CSRF
BugsErrores que provocan comportamiento incorrectoNull dereference, logica incorrecta
Code SmellsProblemas de mantenibilidadFunciones largas, codigo duplicado
Security HotspotsCodigo que requiere revision manualUso de crypto, manejo de datos sensibles
SecretosCredenciales hardcodeadasAPI keys, passwords, tokens

Soporta 30+ lenguajes incluyendo HTML, JavaScript, CSS, TypeScript, Python, Java, etc. Es especialmente relevante para validar codigo generado por IA.

Herramientas involucradas
  • SonarCloud (sonarcloud.io) — Servicio en la nube que analiza el codigo. Gratis hasta 200K lineas.
  • GitHub (github.com) — Donde se aloja el codigo fuente. SonarCloud se conecta a el.
  • analisis-codigo.html — Panel local que muestra los badges y metricas de SonarCloud. Exportable a otros proyectos.
  • OpenCode + MCP — Integracion opcional para que el agente IA verifique codigo automaticamente.

Procedimiento completo paso a paso

FASE 1: Prerequisitos

Necesitas antes de empezar
  • Una cuenta en GitHub (github.com)
  • Git instalado en tu Mac (viene preinstalado en macOS)
  • Un proyecto con codigo que quieras analizar

FASE 2: Subir el proyecto a GitHub (si no lo esta)

SonarCloud necesita que el codigo este en un repositorio de GitHub. Si tu proyecto aun no esta en GitHub:

  1. Abre la Terminal de macOS (Aplicaciones > Utilidades > Terminal)
  2. Navega al proyecto: 
    cd /ruta/a/tu/proyecto
  3. Inicializa Git: 
    git init
git branch -M main
git config user.email "tu-email@ejemplo.com"
git config user.name "Tu Nombre"
  4. Crea un .gitignore (opcional pero recomendado): 
    # Crear archivo .gitignore con exclusiones comunes
echo ".DS_Store
node_modules/
*.log
.env
.scannerwork/" > .gitignore
  5. Haz el primer commit: 
    git add .
git commit -m "Initial commit"
  6. Crea el repositorio en GitHub:
    • Ve a github.com/new
    • Nombre: el nombre de tu proyecto
    • Visibilidad: Public (necesario para SonarCloud gratis)
    • NO marques "Add README" ni ".gitignore"
    • Clic en "Create repository"
  7. Conecta y sube: 
    git remote add origin https://github.com/TU-USUARIO/TU-REPO.git
git push -u origin main

    Cuando pida credenciales:

    • Username: tu usuario de GitHub
    • Password: un Personal Access Token de GitHub (NO tu password). Generalo en: github.com/settings/tokens/new

FASE 3: Crear proyecto en SonarCloud

  1. Ve a sonarcloud.io (no confundir con otros sitios como qt.io o sonarsource.com)
  2. Clic en "Log in" (esquina superior derecha) > elige "GitHub"
  3. Autoriza SonarCloud a acceder a tu GitHub (popup)
  4. En el dashboard, clic en "+" (esquina superior derecha) > "Analyze new project"
  5. Selecciona tu cuenta de GitHub y busca tu repositorio en la lista
  6. Clic en "Set Up" junto al nombre del repo
  7. Elige el plan "Free"
  8. SonarCloud comenzara a analizar automaticamente. Veras el mensaje: "We are analyzing your project. You should see this page refresh in a few moments."
  9. Espera 2-5 minutos. La pagina se refrescara con los resultados.
Analisis automatico
Una vez configurado, SonarCloud analiza automaticamente tu codigo cada vez que haces git push a GitHub. No necesitas ejecutar ningun comando manualmente.

FASE 4: Generar token de SonarCloud

El token es necesario para el panel local y para la integracion con OpenCode.

  1. En SonarCloud, clic en tu avatar (esquina superior derecha) > "My Account"
  2. Ve a la pestana "Security"
  3. En "Generate Token", escribe un nombre (ej: opencode-scanner)
  4. Clic en "Generate"
  5. Copia el token inmediatamente (solo se muestra una vez, empieza por un hash largo)
  6. Guardalo en un lugar seguro. En la terminal puedes configurarlo como variable de entorno: 
    # Temporal (solo esta sesion de terminal)
export SONAR_TOKEN="tu-token-aqui"

# Permanente (anade al final de ~/.zshrc o ~/.bashrc)
echo 'export SONAR_TOKEN="tu-token-aqui"' >> ~/.zshrc

FASE 5: Verificar resultados

Despues del analisis, en SonarCloud veras un dashboard con:

MetricaSignificadoRating ideal
SecurityVulnerabilidades encontradasA (0 vulnerabilidades)
ReliabilityBugs encontradosA (0 bugs)
MaintainabilityCode smells (problemas de limpieza)A
CoveragePorcentaje de codigo cubierto por tests> 80%
DuplicationsPorcentaje de codigo duplicado< 3%
Security HotspotsCodigo que requiere revision manual0

Los ratings van de A (mejor) a E (peor). El Quality Gate es PASSED o FAILED segun se cumplan los umbrales configurados.

URL del dashboard
Tu dashboard esta siempre disponible en: 
https://sonarcloud.io/dashboard?id=TU-ORG_TU-PROYECTO
Ejemplo real: sonarcloud.io/dashboard?id=Jfajeda_PLATAFORMA_OpenCode

Panel local de Analisis (analisis-codigo.html)

El archivo analisis-codigo.html es un panel independiente que muestra los badges y metricas de SonarCloud directamente en tu navegador, sin necesidad de ir a sonarcloud.io. Tiene 5 pestanas:

PestanaContenido
ResumenQuality Gate, badges con metricas principales, ratings
SeguridadDetalle de vulnerabilidades y security hotspots
CalidadBugs, code smells, cobertura, duplicaciones
HistorialRegistro manual de analisis con graficos de tendencia (localStorage)
ConfiguracionGuia paso a paso para configurar SonarCloud y OpenCode

Abrir el panel local

# Opcion 1: Desde la Terminal
open analisis-codigo.html

# Opcion 2: Doble clic en el archivo desde Finder

# Opcion 3: Desde la Plataforma de Seguimiento
# Sidebar > Herramientas > "Analisis de Codigo"

Los badges muestran datos reales automaticamente

El panel obtiene badges SVG directamente de SonarCloud. Si el proyecto es publico, los badges se actualizan automaticamente despues de cada analisis. Los datos que ves en los badges (bugs, vulnerabilities, code smells, ratings) son los mismos que en sonarcloud.io.

Exportar el panel a otro proyecto

El panel analisis-codigo.html esta disenado para ser reutilizable. Para usarlo en otro proyecto:

  1. Copia el archivo analisis-codigo.html al directorio de tu otro proyecto
  2. Abre el archivo con un editor de texto
  3. Busca el bloque SONAR_CONFIG al inicio del <script> (aproximadamente linea 635)
  4. Modifica estos valores: 
    const SONAR_CONFIG = {
  organization: 'TU-ORG-SONARCLOUD',
  sonarUrl: 'https://sonarcloud.io',
  token: '',  // Dejar vacio para badges publicos
  projects: [
    {
      key: 'TU-ORG_NOMBRE-REPO',
      name: 'Nombre visible del proyecto',
    }
  ],
  brand: {
    name: 'TU EMPRESA',
    subtitle: 'Tu descripcion',
    logo: 'https://tu-dominio.com/logo.png',
    primary: '#178DC2',  // Color primario
    accent: '#12A79D',   // Color acento
  }
};
  5. Guarda y abre en el navegador. Los badges se actualizaran automaticamente.
Donde obtener el project key
El project key tiene el formato ORGANIZACION_NOMBRE-REPO. Lo encuentras en:
  • SonarCloud > Tu proyecto > Project Information (sidebar izquierda)
  • O en la URL del dashboard: sonarcloud.io/dashboard?id=ESTE-ES-EL-KEY

Integracion con OpenCode (MCP)

Para que OpenCode pueda consultar los resultados de SonarCloud y verificar codigo automaticamente:

1. Crear opencode.json en la raiz del proyecto

{
  "$schema": "https://opencode.ai/config.json",
  "mcp": {
    "sonarqube": {
      "type": "local",
      "command": ["npx", "-y", "sonarqube-mcp-server@latest"],
      "enabled": true,
      "environment": {
        "SONARQUBE_URL": "https://sonarcloud.io",
        "SONARQUBE_TOKEN": "{env:SONAR_TOKEN}",
        "SONARQUBE_ORGANIZATION": "TU-ORG"
      }
    }
  }
}

2. Crear comandos personalizados (opcional)

Crea la carpeta .opencode/commands/ en tu proyecto y anade:

.opencode/commands/sonar.md

.opencode/commands/sonar-report.md

---
description: Ver reporte de issues de SonarQube
agent: plan
---
Consulta el estado del proyecto en SonarCloud.
1. Obtiene metricas del MCP server
2. Muestra Quality Gate y issues abiertos
3. Sugiere plan de accion por prioridad

Despues puedes ejecutar /sonar o /sonar-report directamente en OpenCode.

Crear sonar-project.properties (analisis manual)

Solo necesario si deseas ejecutar analisis manualmente (sin el automatico de SonarCloud). Crea este archivo en la raiz del proyecto:

# sonar-project.properties

sonar.sources=.
sonar.sourceEncoding=UTF-8
sonar.exclusions=**/node_modules/**,**/dist/**,**/.git/**

Y ejecuta:

export SONAR_TOKEN="tu-token"
npx @sonar/scan
Requisito: Node.js
El comando npx @sonar/scan requiere Node.js 18+. Si no lo tienes instalado, puedes descargar el scanner binario desde docs.sonarsource.com. Sin embargo, con el analisis automatico de SonarCloud no necesitas instalar nada.

Que hacer cuando creas un nuevo proyecto

Cada vez que inicies un nuevo proyecto en OpenCode, actualiza esta lista siguiendo estos pasos:

  1. Crear carpeta en ~/Proyectos/Nombre_Proyecto (sin espacios ni acentos)
  2. Inicializar Git con .gitignore (ver Capitulo 18)
  3. Abrir en OpenCode y ejecutar /init para crear AGENTS.md
  4. Subir a GitHub si quieres backup en la nube (ver Capitulo 17)
  5. Conectar SonarCloud si el proyecto tiene codigo analizable (ver Capitulo 17)
  6. Actualizar este capitulo anadiendo el nuevo proyecto a las tablas

Que hacer para subir un proyecto existente a GitHub

Para los 7 proyectos que todavia no estan en GitHub:

  1. Ve a github.com/new
  2. Nombre del repositorio: el mismo nombre de la carpeta (ej: legislacion_monitor)
  3. Visibilidad: Public (gratis con SonarCloud) o Private
  4. NO marcar "Add README" ni ".gitignore" (ya los tenemos)
  5. Clic en "Create repository"
  6. En la Terminal: 
    cd ~/Proyectos/nombre_proyecto
git remote add origin https://github.com/Jfajeda/nombre_proyecto.git
git push -u origin main

    Username: Jfajeda — Password: token de GitHub (ver generar token)
